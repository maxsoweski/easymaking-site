The two-phase dance

In the previous lesson the browser did all three things every frame: re-wrap the text (analyze), compute the line break positions (fit), and paint the result (render). Pretext splits the first two out of the browser and moves them to JavaScript, where you control when they happen.

The split is simple:

  • prepare(text, font) — run once. Segments the text, asks the canvas for every segment's width, caches the result. Comparatively expensive.
  • layout(prepared, width, lineHeight) — run many times. Pure arithmetic on the cached widths. Returns { height, lineCount }.

Mechanism

When the paragraph count changes, we call prepare() on each paragraph's text with a fixed font descriptor. Pretext segments the text, probes each segment's width using the browser's own canvas font engine, and stores the result in an opaque handle.

When the width changes, we call layout() on every prepared handle with the new width. Pretext walks the cached segments and applies its word-wrap rules entirely in arithmetic — no DOM reads. The handle is reusable across every width you'll ever ask about.

The word-wrap rules it reproduces are the browser's: Unicode line-break classes, CJK keep-all, bidirectional text, soft hyphens, emoji grapheme clusters. The canvas measurement is the ground truth, so the line breaks match what the browser would have produced.

Application

Every effect the previous lesson listed as blocked becomes tractable:

  • A pull-quote whose font size breathes with the viewport — you call layout() on each pointermove; no reflow.
  • A chat bubble that knows its height before insertion — prepare() the message string, layout() at the bubble's max width, read the height, insert with it pre-set.
  • A masonry grid that reflows continuously — heights are known in JS, placement is arithmetic.
  • Text that hugs a draggable image — layoutNextLineRange() with per-line width, the width being a function of the image's current y.

The shape of the unlock is the same each time. You stop asking the DOM for measurements. You do the measurement yourself, once, with a tool that knows what the browser knows.

Direct Claude

These are the phrases that point at the two-phase pattern when you're briefing a build:

"pre-measure the messages" → call prepare() on each string up front, cache the handle "reflow without measuring" → call layout(handle, width, lineHeight) per width; no DOM reads "heights known at scroll time" → virtualize using the height Pretext returns, not DOM-measured "make it breathe on resize" → bind layout() to the resize observer; animate the computed height
The rest of Arc 1 was here. The rest of the course shows what this two-phase pattern unlocks — starting with the simplest question: given text and width, how tall?
prepare once, layout many 
import { prepare, layout } from '@chenglou/pretext';

// Phase 1: analyze — run once per text string.
const handles = paragraphs.map(p =>
  prepare(p.textContent, "400 20px 'EB Garamond', serif")
);

// Phase 2: fit — run each frame. Pure arithmetic.
function layoutFrame() {
  handles.forEach((h, i) => {
    const { height } = layout(h, currentWidth, 20 * 1.55);
    paragraphs[i].style.height = height + 'px';
  });
  fps.tick();
  requestAnimationFrame(layoutFrame);
}
