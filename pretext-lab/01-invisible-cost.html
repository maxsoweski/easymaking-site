The invisible cost

When the browser lays out text, it does real work — and most of it is hidden. getBoundingClientRect() looks like a cheap read, but reading a measured value while the renderer has pending work flushes the entire pipeline. Do that sixty times a second across a few dozen paragraphs and the page starts to stutter.

Mechanism

Whenever the width changes, every paragraph must be re-wrapped. The browser already knows how to do this — it's what the renderer does all day. The problem is when your JavaScript also wants to know the result.

Calling element.getBoundingClientRect() is the canonical way to ask the browser "how tall did that end up?" But to give you a correct answer, the browser must finish any pending layout first. That's a synchronous flush. Inside a loop that both writes style and reads a measured value, each iteration triggers its own layout. The cost is per-iteration, not amortized.

This is called layout thrashing. It's the number-one cause of sluggish sliders, flickering pull-quotes, and chat bubbles that jump a pixel on mount.

Application

Most of what makes a text-heavy page feel alive is downstream of this cost. Consider the effects that should be possible but rarely are:

  • A pull-quote whose font size breathes with the viewport, with no reflow flash.
  • A chat bubble that knows its height before it is inserted, so nothing else on screen moves when it arrives.
  • A masonry grid where text cards reflow to fill gaps as the window resizes, continuously, without a jank frame.
  • Text that hugs a draggable image — the text stays pinned while the image moves, re-routing around the new obstacle at 60 fps.

The browser's text engine is capable of all of this. It's the measurement step — reading what the engine decided — that blocks us.

Direct Claude

If you want any of the effects from the Application list, naming the problem clearly helps. These are phrasings that point at the fix the next lesson introduces:

"make the slider settle instantly on release" → stop reading back from the DOM inside the input handler "no flash when the card mounts" → know the card's height before you insert it "live drag should feel like moving paper" → avoid getBoundingClientRect in pointermove "responsive sweep with no stutter" → pre-measure once, then lay out by arithmetic

// Per-frame layout thrash: write a style, then read a measured value.
// Each iteration forces the browser to flush layout.
function thrashFrame() {
  const paragraphs = textRegion.querySelectorAll('p');
  paragraphs.forEach((p, i) => {
    p.style.paddingLeft = (i % 2) + 'px';     // write
    void p.getBoundingClientRect().height;    // read — forces layout
  });
  fps.tick();
  requestAnimationFrame(thrashFrame);
}
